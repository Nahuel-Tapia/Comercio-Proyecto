Estructura Técnica del Backend

Guía de arquitectura, funcionamiento de archivos y patrones de diseño implementados.

Estructura de Arquitectura Limpia (Clean Architecture)

El proyecto está desarrollado utilizando los principios de **Clean Architecture**, lo que asegura un alto nivel de desacoplamiento tecnológico. Las dependencias van de afuera hacia adentro: las capas externas (Bases de datos, Controladores API) conocen las capas internas (Reglas de Negocio), pero el núcleo central (Dominio) no tiene dependencia de nada exterior.

Ventaja Clave para la Defensa del Proyecto
Si el día de mañana se decide cambiar la base de datos de SQLite a PostgreSQL o SQL Server, solo se modifica la **Capa de Infraestructura**. Todo el código de negocio del **Dominio** y los casos de uso de **Aplicación** permanecen intactos.

1. Capa ECommerce.Domain (El Corazón del Negocio)

Sigue los principios de Domain-Driven Design (DDD). Su principal característica es que las entidades controlan su propio comportamiento interno y no permiten estados inválidos.

Value Objects (Objetos de Valor)

No tienen una identidad única (un Id). Se definen por la igualdad de sus propiedades y son inmutables.

  • Money.cs: Encapsula una cantidad (decimal Amount) y la moneda (string Currency). Valida que no existan valores negativos y maneja sumas aritméticas seguras de dinero.
  • Email.cs: Valida que toda cadena asignada a un correo electrónico cumpla con el formato RegExp estándar en su constructor. Si es incorrecto, lanza un error de forma inmediata.
  • ProductName.cs: Valida que el nombre de un producto sea obligatorio y no supere los 200 caracteres de longitud.

Entities (Entidades)

Tienen un identificador único (Id) y representan los objetos clave del negocio con comportamiento dinámico.

  • Product.cs: Representa un producto en venta. Expone el método Reserve(quantity) que descuenta el stock de manera controlada o lanza una excepción específica si no hay disponibilidad.
  • Order.cs: Representa una compra. Contiene el método AddItem(product, quantity). Este método se asegura de que el producto sea válido, restringe el stock del mismo e incrementa el total de la orden sumando los objetos Money.
  • OrderItem.cs: Detalle individual (línea de factura) de un producto agregado a la orden con su precio unitario y cantidad.
  • User.cs: Usuario del sistema con su rol de seguridad (Admin o User) y su clave encriptada mediante hashing seguro.

Domain Exceptions

  • DomainRuleException.cs: Excepción base para cualquier violación de las reglas de negocio del dominio.
  • InsufficientStockException.cs: Se lanza cuando se intenta realizar un pedido que excede el stock disponible, previniendo compras inconsistentes.

2. Capa ECommerce.Application (Casos de Uso)

Esta capa implementa el patrón **CQRS** (Command Query Responsibility Segregation). Las operaciones de lectura (Consultas/Queries) están completamente separadas de las operaciones de escritura (Comandos/Commands). Para lograr este desacoplamiento se utiliza la librería **MediatR**.

Estructura de Features (UseCases)

UseCases/
  ├── Products/
  │   ├── Commands/
  │   │   ├── CreateProductCommand.cs // Crea el producto y valida reglas
  │   │   └── DeleteProductCommand.cs // Elimina por Id (solo Administradores)
  │   ├── Queries/
  │   │   ├── GetAllProductsQuery.cs // Lista todo el catálogo
  │   │   └── GetProductByIdQuery.cs // Busca un producto individual
  │   └── ProductMapper.cs // Conversiones rápidas Entity <-> DTO
  └── Orders/
      └── Commands/
          ├── CreateOrderCommand.cs // Compra y descuenta stock en una transacción
          └── OrderItemDto.cs // Objeto de transporte para los items

Pipeline Behaviors: Filtro Automático de Validaciones

ValidationBehavior.cs: En lugar de escribir comprobaciones manuales de validación dentro de cada controlador HTTP o de cada comando, el proyecto cuenta con un interceptor global de MediatR. Cada vez que viaja un comando, este interceptor busca si existe un validador escrito en FluentValidation (como CreateProductValidator.cs), lo ejecuta de forma automática y cancela la operación antes de llegar al controlador si hay errores.

Mapeadores Limpios (Mappers)

ProductMapper.cs: Realiza el mapeo directo campo a campo entre las entidades del dominio y los DTOs que se retornan al exterior de forma manual. Esto evita dependencias pesadas de librerías como AutoMapper y mantiene la velocidad de procesamiento al máximo.

3. Capa ECommerce.Infrastructure (Tecnología y Persistencia)

Es la capa técnica del proyecto. Implementa los contratos definidos en la capa de aplicación interactuando con librerías del sistema, bases de datos y la red.

Persistencia con Entity Framework Core

  • ApplicationDbContext.cs: Declara las tablas de base de datos (DbSet) y aplica configuraciones específicas usando Fluent API en lugar de decoradores en las entidades (lo que mantendría acoplado el dominio a la base de datos).
  • Configurations (UserConfiguration.cs, etc.): Permite mapear los Value Objects del dominio (como Money) a tipos de datos planos soportados por SQLite (por ejemplo, guardando el precio como un decimal simple y reconstruyendo el objeto al recuperarlo gracias al método .HasConversion()). Además, aquí se ejecuta la creación inicial de usuarios mediante Seed Data.
  • SQLite: Motor de base de datos SQL local ligero y veloz, ideal para ambientes de demostración y pruebas sin necesidad de un servidor externo.

Seguridad y Autenticación

  • JwtTokenService.cs: Se encarga de generar los tokens JSON Web Token (JWT) tras un inicio de sesión exitoso. Los tokens van firmados digitalmente mediante una clave secreta configurada en el servidor y portan los roles del usuario.
  • BCrypt: Herramienta estándar de encriptación utilizada para aplicar Hash seguro a las contraseñas de los usuarios persistidos en base de datos. Ninguna contraseña es legible en texto plano en la base de datos SQLite.

Manejador Global de Errores

  • GlobalExceptionHandler.cs: Es un Middleware de ASP.NET Core que actúa como red de seguridad para la API. Si ocurre un error inesperado (o una excepción de negocio como InsufficientStockException), este middleware la atrapa, detiene la caída de la aplicación, asigna el código HTTP adecuado (400 Bad Request, 404 Not Found, 401 Unauthorized) y devuelve un formato estándar legible para los clientes: RFC 7807 (Problem Details).

4. Capa ECommerce.Api (La Entrada del Sistema)

Es una API Web REST construida sobre ASP.NET Core. No contiene lógica de negocio y solo funciona como capa de despacho y serialización JSON.

Controladores HTTP

  • AuthController.cs: Expone el endpoint /api/auth/login. Verifica el usuario/contraseña usando BCrypt y delega la emisión del token al JwtTokenService.
  • ProductsController.cs: Expone endpoints CRUD para el catálogo. Cuenta con restricciones de roles: solo los administradores con token válido ([Authorize(Roles = "Admin")]) pueden crear o eliminar artículos.
  • OrdersController.cs: Expone la ruta /api/orders para enviar una solicitud de compra de productos de manera autenticada.

5. Pruebas y Aseguramiento de Calidad

El código cuenta con una sólida estrategia de pruebas automatizadas:

Aseguramiento mediante Dos Tipos de Pruebas
Pruebas Unitarias: Validan la consistencia de las reglas (ej. que no se puedan crear importes de dinero negativos en Money).
Pruebas de Integración (E2E): Simulan peticiones HTTP reales a la API usando un script automatizado para verificar el flujo completo.

Script de Pruebas de Integración E2E (test_api.ps1)

El archivo de prueba de integración automatizada realiza el siguiente flujo de extremo a extremo:

  1. Inicia sesión con la cuenta de Usuario Regular y comprueba la obtención de un token JWT.
  2. Inicia sesión con la cuenta de Administrador para adquirir permisos elevados.
  3. Registra un producto de prueba interactuando con el endpoint POST autenticado.
  4. Prueba los controles de acceso: el usuario regular intenta borrar el producto y el servidor debe responder con un error **403 Forbidden** (Acceso Denegado).
  5. El administrador realiza el borrado del producto exitosamente.
  6. Crea un pedido con múltiples ítems, validando que el motor de persistencia SQLite procese la transacción, actualice los inventarios correspondientes y responda con **201 Created**.
